01 · Executive Message

核心结论：这不是读材料工具，而是标签治理流程。

它把「证据、字段、结论、归档」串成闭环，让基金赛道审核从一次性判断变成团队资产。

A

证据可追溯

区分 P1 尽调材料和 P2 当前字段，建议必须说明来源与强度。

B

结论可复核

统一使用一致、需补充、需修正、需复核补充、人工复核等口径。

C

报告可沉淀

单基金报告进入共享报告库，后续可查询、复用和对比。

D

流程可迁移

workspace 接入、规则拆分、词表脚本、strict 校验都可迁移到其他 Skill。

02 · Why Build

建设背景：基金标签是基础数据，但维护天然滞后。

核心矛盾不是没有材料，而是材料、系统字段和人工判断之间缺少稳定连接。

问题 典型表现 业务影响 Skill 的处理方式
字段更新滞后 管理人策略变化，`zmdata` 仍是旧赛道。 筛选和对比失真。 用最新 P1 材料对照当前字段。
标签颗粒度不足 只填「股票多头」或「CTA」，缺少细分赛道。 无法精细化研究。 通过受控词表验证候选 `raceName`。
同系列口径不一致 同管理人不同产品字段不统一。 数据治理成本上升。 区分主审记录和参考记录。
人工判断难沉淀 每次重新翻材料、重新写结论。 经验无法复用。 报告归档并支持历史复用。
03 · Audit Scope

审核边界：字段治理和策略画像分开。

研究信息可以丰富展示，但真正进入字段审核的对象必须受控。

字段 含义 审核重点
strategyType 一级策略 是否属于正确大类，如 CTA、股票多头、市场中性。
raceName 细分赛道 是否过宽、缺失或与材料冲突。
fundType 波动层级 是否与风险评级、杠杆、基准和净值特征匹配。

边界原则

策略画像服务于研究理解，字段审核服务于数据治理。`labelIDs` 只展示，不纳入人工字段审核。

Display L1/L2/L3、资产覆盖、子策略、风控、容量、换手率。
Audit 只判断系统中需要治理的结构化字段。
04 · Operating Model

工作原理：7 步完成从证据到归档的闭环。

流程稳定比单次回答漂亮更重要。每一步都服务于可追溯和可复用。

1

检查环境

确认尽调材料盘和共享报告库可访问。

2

查历史报告

未要求刷新时，优先复用已有单基金报告。

3

查 zmdata

获取基金 ID、代码、管理人和当前字段。

4

定位证据

读取尽调笔记、投决会纪要和路演材料。

5

构建画像

抽取赛道、收益来源、风险画像和标签映射。

6

字段审核

对照受控词表和证据强度给出结论。

7

校验归档

通过 strict 校验后同步到共享报告库。

P1

尽调材料

用于判断真实策略画像和标签建议，包括尽调笔记、尽调问卷、投决会纪要、路演材料和营销材料。

P2

zmdata 当前字段

作为被审核的系统字段基线。P2 不替代 P1，只说明当前结构化数据长什么样。

05 · Efficiency Playbook

建设提效技巧：效率来自工作流设计，而不只是模型能力。

这些做法可以迁移到管理人画像、产品线分类、投后质检和批量标签清洗等场景。

网盘加入 workspace

将 `W:` / `Z:` 尽调材料和 `Y:` 报告库接入工作区，用原生工具读文件，替代重型 SMB 管道。

迁移经验：共享资料先接入 workspace。

规则拆分

把主流程、证据规则、输出模板、保存流程拆成独立 reference 文件，降低上下文噪音。

迁移经验：复杂 Skill 不写成大提示词。

固定缓冲区保存

先写 `latest_report.md`，再校验、同步，避开中文路径和命令行内嵌字符串的编码风险。

迁移经验：中文长文档走缓冲文件。

Strict 校验前置

用脚本检查章节、证据等级、字段行、推论边界和表格格式，模型不能绕过质检。

迁移经验：共享产物必须机器校验。

历史报告复用

先查历史产物，未要求刷新就复用；多基金先预检，避免重复审计。

迁移经验：研究工作先查缓存。

受控词表脚本

用 `_vocabulary.py` 查询标签、波动层级和 label ID，避免生成系统中不存在的候选值。

迁移经验：内部枚举配轻量 CLI。

长文档定向抽取

PDF 按页码、关键词和上下文页抽取，DOCX 限长提取，减少材料噪音。

迁移经验：先取证，再判断。

主审 / 参考分离

多记录命中时，明确主审对象和同系列参考对象，避免字段互相污染。

迁移经验：多记录业务显式定主对象。

证据源只读

尽调材料只允许读取、搜索和抽取，派生报告写入共享报告库或缓冲区。

迁移经验：原始证据和派生成果分离。

终端输出隔离

盘符探测、查询、枚举、词表、保存分别执行，输出更干净，排错更快。

迁移经验：终端输出也是上下文资产。
06 · How To Use

使用方法：默认复用历史，明确刷新才重跑。

这样能同时兼顾效率和时效性，避免团队在同一基金上反复做相同审核。

# 首次环境检查
Test-Path "W:\投顾信息（PPT、尽调反馈表等）"
Test-Path "Z:\投顾信息（PPT、尽调反馈表等）"
Test-Path "Y:\投顾管理人研究\fund-track-tag-audit"

# 单基金审核
用 fund-track-tag-audit 审核一下 <基金名称> 的策略标签。

# 重新审核
重新审核 <基金名称>，以当前 zmdata 和最新尽调材料为准。

# 多基金审核
请审核以下基金的赛道标签：A 基金、B 基金、C 基金。
默认路径有历史报告且未要求刷新，直接返回历史报告。
刷新路径用户明确说重新审核、最新、当前，才跑完整流程。
多基金路径先预检，已有报告复用，未覆盖基金新审。
阅读重点优先看结论摘要、字段审核、关键证据和口径说明。
07 · Team Adoption

团队协作：不同角色看不同层次。

同一份报告既是研究证据，也是数据治理依据，还是 Skill 后续维护的反馈入口。

投研 / FOF 研究

  • 入池前标签核验。
  • 投后复盘时更新策略画像。
  • 同系列产品口径统一。
  • 向数据团队提交字段修正依据。

数据治理团队

  • 批量检查字段缺失。
  • 识别 `raceName` 过宽产品。
  • 建立 `fundType` 人工复核清单。
  • 确认建议值是否通过受控词表校验。

Skill 维护者

  • 维护审核规则和报告模板。
  • 更新词表解析脚本。
  • 补充 strict 质检规则。
  • 监控归档路径和保存流程稳定性。
08 · Closing

方法论价值：从单次 AI 辅助，走向团队级数据治理。

这套方法可以继续迁移到管理人画像更新、产品线分类、策略变更追踪、投后报告质检和批量标签清洗。

它把「读材料」升级为「证据化审核」，把「一次性回答」升级为「可归档报告」，把「个人经验」升级为「团队可复用流程」。

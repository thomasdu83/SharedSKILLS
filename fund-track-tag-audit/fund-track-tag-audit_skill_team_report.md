# fund-track-tag-audit Skill 建设报告

生成时间：2026-06-01  
分享对象：投研、FOF 研究、数据治理、运营尽调、AI 工作流维护人员  
分享目标：说明这个 Skill 解决什么问题、如何工作、如何使用，以及建设过程中哪些技巧显著提高了效率。

---

## 1. 核心结论

`fund-track-tag-audit` 是一个用于审核基金赛道、标签和波动层级的团队级 AI 工作流。它把投顾尽调材料中的新信息，与 `zmdata` 中已有的结构化字段进行对照，判断 `strategyType`、`raceName`、`fundType` 是否一致、缺失或需要修正。

这个 Skill 的重点不是「让 AI 读材料后随手给标签」，而是把基金标签治理变成一套可重复、可验证、可归档的流程：

- **证据可追溯**：区分尽调材料和当前系统字段，所有建议都要说明依据。
- **结论可复核**：字段级结论统一使用固定口径，避免每次表达不一致。
- **报告可沉淀**：单基金报告自动进入共享报告库，后续可复用、可比对。
- **流程可迁移**：里面的工作区接入、受控词表、报告校验、历史复用等技巧，也适合复制到其他投研数据治理类 Skill。

---

## 2. 为什么要建设这个 Skill

基金赛道和标签是投研筛选、产品比较、组合配置、投后跟踪的基础字段。现实工作中，这类字段经常出现 4 个问题：

| 问题 | 典型表现 | 影响 |
|---|---|---|
| 字段更新滞后 | 管理人策略已经变化，但 `zmdata` 仍是旧赛道 | 筛选和对比失真 |
| 标签颗粒度不足 | 只填了「股票多头」或「CTA」，缺少细分赛道 | 无法做精细化研究 |
| 同系列口径不一致 | 同一管理人不同产品的 `raceName` 或 `fundType` 不统一 | 数据治理成本上升 |
| 人工判断难沉淀 | 每次都重新翻材料、重新写结论 | 经验无法复用 |

因此，这个 Skill 的建设目标很明确：  
**用尽调证据校验结构化标签，用固定流程沉淀审核结论，用共享报告库减少重复劳动。**

## 3. 它审核什么，不审核什么

### 3.1 重点审核字段

| 字段 | 含义 | 审核重点 |
|---|---|---|
| `strategyType` | 一级策略 | 是否属于正确大类，如 CTA、股票多头、市场中性 |
| `raceName` | 细分赛道 | 是否过宽、缺失或与材料冲突，如是否应细化为「300指增」 |
| `fundType` | 波动层级 | 是否与材料中的风险评级、杠杆、基准和净值特征匹配 |

### 3.2 只展示、不强行审核的信息

尽调材料里会有大量有价值的信息，例如投资范围、子策略、风控流程、资产覆盖、容量、换手率等。Skill 会把这些内容整理成「策略画像」，但不会把所有信息都强行转成结构化字段。

这个边界很重要：  
**策略画像服务于研究理解，字段审核服务于数据治理。两者不能混在一起。**

### 3.3 不纳入审核的字段

`labelIDs` 只作为展示信息，不纳入字段审核。它是 `zmdata` 系统内置多层标签树，随基金数据自动同步，不是人工维护字段。

## 4. 工作原理

### 4.1 一条主线

这个 Skill 的主流程可以概括为 7 步：

1. **检查环境**：确认尽调材料盘和共享报告库可访问。
2. **查历史报告**：如果已有单基金报告且用户未要求刷新，优先复用。
3. **查 `zmdata` 当前字段**：获取基金 ID、代码、管理人、当前赛道和波动层级。
4. **定位尽调证据**：在管理人文件夹下读取尽调笔记、投决会纪要、路演材料等。
5. **构建策略画像**：抽取一级赛道、细分赛道、收益来源、风险画像和标签映射。
6. **逐字段审核**：对照受控词表和证据强度，判断是否一致、补充、修正或复核。
7. **校验并归档报告**：报告先通过 strict 校验，再同步到共享报告库。

### 4.2 证据分层

Skill 只使用 P1/P2 两级证据：

| 级别 | 来源 | 用途 |
|---|---|---|
| P1 | 尽调材料，包括尽调笔记、尽调问卷、投决会纪要、路演材料、营销材料 | 判断真实策略画像和标签建议 |
| P2 | `zmdata` 当前字段 | 作为被审核的系统字段基线 |

如果多份 P1 材料一致，证据强度可以为「高」；如果只有单份 P1 或材料偏弱，通常为「中」；如果只是同系列间接推断，则需要降级并提示人工复核。

### 4.3 受控词表校验

给出 `raceName` 或 `fundType` 建议前，Skill 会先查内部受控词表，确认候选标签真实存在。对应工具是：

- `_vocabulary.py search <关键词>`
- `_vocabulary.py lookup <label_id>`
- `_vocabulary.py strategy-types`
- `_vocabulary.py race-names`
- `_vocabulary.py risk-labels`

这个机制可以防止 AI 生成「看起来合理，但系统里没有」的标签。

### 4.4 报告结构

单基金报告固定为 6 段：

1. 结论摘要
2. 基金记录（zmdata）
3. 字段审核
4. 策略画像
5. 关键证据
6. 口径说明

报告不是自由发挥，而是标准产物。这样便于后续复用、对比、质检和批量治理。

## 5. 建设过程中提高效率的关键技巧

这一部分是最值得复用的经验。很多效率提升并不来自更复杂的模型，而来自把工作流设计得更顺。

### 5.1 把网盘加入 workspace，减少远程文件处理成本

最关键的提效动作，是把两个网络盘加入当前 workspace：

- `W:\` 或 `Z:\`：投顾尽调材料源。
- `Y:\`：共享报告库。

这样做之后，AI 不需要通过一套复杂的 Python SMB 管道去远程拉取文件，而是可以使用 IDE 原生能力进行目录探测、文件读取和文本搜索。  
早期方案依赖较重的 SMB 脚本，路径、编码、权限、超时都更容易出问题。改成「网盘加入 workspace + 原生工具读取」后，流程更轻，定位问题也更快。

可迁移经验：  
**凡是资料本身已经存在于共享盘，优先把共享盘接入 workspace，而不是先写一套重型远程文件访问系统。**

### 5.2 把 Skill 拆成主流程和参考规则

当前 Skill 没有把所有规则都塞进一个超长提示词，而是拆成：

| 文件 | 作用 |
|---|---|
| `SKILL.md` | 定义主流程和强约束 |
| `references/source-priority.md` | 定义证据优先级 |
| `references/decision-rules.md` | 定义字段审核规则 |
| `references/output-templates.md` | 定义报告模板 |
| `references/save-and-sync.md` | 定义保存和归档流程 |

这样做有两个好处：

- 模型只在需要时读取相关规则，减少上下文噪音。
- 规则可以单独维护，不必每次改动都重写整个 Skill。

可迁移经验：  
**复杂 Skill 不要写成一整篇大提示词，应拆成「主流程 + 决策规则 + 输出模板 + 工具脚本」。**

### 5.3 用固定缓冲区解决中文报告保存问题

报告保存没有使用 `python -c`、PowerShell here-string 或管道写中文，因为这些方式在多层沙盒和中文路径下容易出现转义损坏或乱码。

最终采用固定链路：

1. 写入 `latest_report.md`。
2. 运行 `validate_audit_report.py --strict`。
3. 运行 `sync_to_y.py` 同步到 Y 盘。
4. 验证目标文件可读。

这个设计看似多一步，但非常稳。它把「生成报告」「校验报告」「归档报告」拆开，任何一步失败都能定位。

可迁移经验：  
**中文长文档不要依赖命令行内嵌字符串保存，优先使用固定缓冲文件和专用同步脚本。**

### 5.4 用 strict 校验把质量控制前置

`validate_audit_report.py --strict` 会检查：

- 是否有 6 个固定章节。
- 表格是否只出现在允许章节。
- 证据强度是否只使用「高 / 中 / 低」。
- 关键证据是否只使用 P1/P2。
- 三个必审字段是否都有结论。
- 同系列推断时 `fundType` 是否足够保守。
- 口径说明是否写明「推论边界风险」。

这相当于把人工审稿中的格式问题、证据问题和边界问题提前自动拦截。模型可以写报告，但不能绕过质检。

可迁移经验：  
**只要输出要进入团队共享库，就应该有机器校验，而不是只靠人工肉眼检查。**

### 5.5 历史报告复用，避免重复审计

`find_saved_report.py` 和 `plan_multi_fund_audit.py` 解决了一个很实际的问题：很多基金已经审过，不需要每次都重跑。

默认策略是：

- 单基金：有历史报告且用户未要求刷新，就复用。
- 多基金：先预检，已有报告的复用，缺失的再新审。
- 用户明确说「重新审核」「最新」「当前」时，才跳过历史复用。

可迁移经验：  
**重复性研究工作要先查缓存和历史产物，再决定是否重新计算。**

### 5.6 用受控词表脚本降低人工查标签成本

`_vocabulary.py` 把标签查询变成命令行工具。审核时不需要人工翻表，也不需要让模型凭记忆判断标签是否存在。

它解决了 3 个问题：

- 快速确认候选标签是否合法。
- 批量解析 `labelIDs`。
- 区分一级策略、细分赛道和波动层级。

可迁移经验：  
**凡是有内部枚举、标签树、字典表的业务，都应给 Skill 配一个轻量查询工具。**

### 5.7 用专用抽取脚本控制材料噪音

PDF 和 DOCX 材料通常很长，如果全文丢给模型，会浪费上下文，也容易抓错重点。Skill 使用：

- `extract_pdf_pages.py`：支持按首页、页码范围、关键词和上下文页抽取。
- `extract_docx.py`：提取段落和表格文本，并限制最大长度。

这让证据读取更像「定向取证」，而不是「整份材料倒进去」。

可迁移经验：  
**长文档处理要先做定向抽取，再让模型判断；不要默认全文投喂。**

### 5.8 主审记录和参考记录分离

基金查询经常会命中多条记录，例如尽调记录、投后子基金记录、匿名记录、同系列产品记录。Skill 要求报告写清楚：

- 哪一条是主审记录。
- 哪些只是参考记录。
- 为什么这样选择。
- 参考记录不能等同于主记录字段值。

这个规则有效避免了「同系列产品字段互相污染」。

可迁移经验：  
**多记录命中的业务场景，一定要显式区分主对象和参考对象。**

### 5.9 证据源只读，派生成果另存

尽调材料目录被定义为只读证据源。Skill 只能读取、搜索和抽取文本，不能移动、删除、重命名或覆盖原文件。所有报告都写入共享报告库或本地缓冲区。

可迁移经验：  
**原始证据源和派生成果必须分离，尤其是投研、尽调、合规类资料。**

### 5.10 终端按任务隔离，减少排错成本

Skill 要求不同性质的命令分开执行，例如盘符探测、`zmdata` 查询、文件枚举、词汇表查询、报告保存不要混在同一个终端输出里。

这样做的好处是排错非常清楚：哪个阶段失败、失败输出是什么，一眼能看到。

可迁移经验：  
**AI 工作流里，终端输出也是上下文资产。保持输出干净，比少开一个终端更重要。**

## 6. 如何使用

### 6.1 首次使用

首次使用前确认：

```powershell
Test-Path "W:\投顾信息（PPT、尽调反馈表等）"
Test-Path "Z:\投顾信息（PPT、尽调反馈表等）"
Test-Path "Y:\投顾管理人研究\fund-track-tag-audit"
python -c "import zmdata; print(zmdata.__version__)"
```

如果尽调材料盘或共享报告库不可达，Skill 会停止，不会继续生成不完整结论。

### 6.2 单基金审核

常用指令：

```text
用 fund-track-tag-audit 审核一下 <基金名称> 的策略标签。
```

如果要基于最新材料重跑：

```text
重新审核 <基金名称>，以当前 zmdata 和最新尽调材料为准。
```

默认行为：

- 有历史报告且未要求刷新：返回历史报告。
- 无历史报告或要求刷新：执行完整审核。
- 完整审核后：保存单基金 Markdown 报告，并返回摘要和路径。

### 6.3 多基金审核

常用指令：

```text
请审核以下基金的赛道标签：A 基金、B 基金、C 基金。
```

默认行为：

- 先逐只基金查历史报告。
- 已有历史报告的默认复用。
- 没有历史报告的执行新审核。
- 用户要求「全部重新审核」时，才全部重跑。

### 6.4 阅读报告时看哪里

建议优先看 4 个位置：

| 章节 | 重点 |
|---|---|
| 结论摘要 | 是否一致、是否需要补充或修正、证据强度如何 |
| 字段审核 | 三个字段当前值、结论、建议值和说明 |
| 关键证据 | 结论来自直接 P1，还是同系列间接推断 |
| 口径说明 | 是否存在推论边界、基准差异、杠杆差异或人工确认点 |

## 7. 团队使用建议

### 7.1 投研和 FOF 研究

适合用于：

- 入池前标签核验。
- 投后复盘时更新策略画像。
- 同系列产品口径统一。
- 向数据团队提出字段修正建议前整理证据。

建议请求时尽量提供基金全称或基金代码，并说明是否需要「重新审核」。

### 7.2 数据治理团队

适合用于：

- 批量检查字段缺失。
- 找出 `raceName` 过宽的产品。
- 建立 `fundType` 人工复核清单。
- 把审核报告作为字段修正依据。

重点关注报告里的建议值是否通过受控词表校验，以及结论是否来自直接证据。

### 7.3 Skill 维护者

重点维护：

- `decision-rules.md`：审核规则是否符合最新业务口径。
- `output-templates.md`：报告模板是否仍适合团队阅读。
- `_vocabulary.py`：词表解析是否覆盖最新标签。
- `validate_audit_report.py`：质检规则是否需要新增。
- `save-and-sync.md`：归档路径和保存流程是否仍稳定。

## 8. 价值总结

`fund-track-tag-audit` 的价值可以归纳为三句话：

- 它把「读材料」升级为「证据化审核」。
- 它把「一次性回答」升级为「可归档报告」。
- 它把「个人经验」升级为「团队可复用的标签治理流程」。

建设这个 Skill 的经验也说明：高效的 AI 工作流不一定依赖复杂系统，很多时候来自几个朴素但关键的设计选择：把共享盘接入 workspace、把规则拆成可维护文件、用脚本处理受控词表和长文档、用 strict 校验守住质量、用历史报告复用减少重复劳动。

这套方法后续可以继续迁移到更多投研数据治理场景，例如管理人画像更新、产品线分类、策略变更追踪、投后报告质检和批量标签清洗。

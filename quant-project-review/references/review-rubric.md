# Quant Project Review Rubric

本文件定义如何判断“完成了多少”和“是否走偏”。

## 1. 初始目标重建

优先从以下来源重建初始目标：

1. 用户最早的需求描述
2. 书面计划或规格说明
3. `project.yaml`
4. 主输出物定义
5. 当前目录结构和运行入口

如果目标仍不明确，输出时必须标注：

- `初始目标为推断版本`
- `置信度：高 / 中 / 低`

## 2. 项目阶段判断

### `idea`

- 只有草稿、笔记、零散探索
- 还没有稳定的项目入口或标准输出

### `research`

- 已有可重复研究代码
- 有研究产物，但尚未形成稳定候选输出

### `candidate`

- 已固化关键输入输出
- 已有候选级快照或候选入口
- 开始关注稳定性、验证、契约

### `production`

- 已能稳定输出用于决策的信号、评分、排序或组合影响
- 有正式运行链路或接入工作流

### `monitor_only`

- 主要输出状态、告警、变化和复核线索
- 不直接作为投资决策主输出

## 3. 四个维度的判断口径

### 目标完成度

看的是“初始承诺”而不是“总工作量”。

高：
- 初始目标中的核心能力基本齐全
- 剩余工作主要是增强项或边缘项

中：
- 主目标方向没错，但只完成了一部分核心能力

低：
- 大部分工作仍停留在准备层、探索层或配套层

### 主链路完成度

主链路通常指：

`数据输入 -> 清洗/加工 -> 特征/因子/信号 -> 输出/快照/报告 -> 消费方`

高：
- 主链路已经打通并可重复运行

中：
- 中间核心逻辑已成型，但闭环尚未完全打通

低：
- 仍停留在局部模块、实验片段或孤立脚本

### 验证完成度

看是否具备这些证据：

- 单元测试 / 集成测试
- 回测或研究验证
- 样本外或稳定性检查
- 字段口径、时间对齐、缺失值处理确认
- 输出契约或结果检查

高：
- 主链路关键点已有验证

中：
- 有局部验证，但覆盖不完整

低：
- 基本靠人工印象或一次性试跑

### 偏航程度

看最近一段工作是否还服务初始目标。

低偏航：
- 最近工作大多直接推进主链路或主输出物

轻微偏航：
- 有一部分工作在做扩展，但主目标仍在推进

明显偏航：
- 最近投入主要用于非核心扩展、装饰层、外围工具或过早泛化

## 4. 偏航信号

以下现象通常说明偏航风险升高：

- 项目配套越来越多，但主输出物仍未定义清楚
- 前端、报表、工具层做了很多，核心策略或模型闭环没打通
- 写了大量通用框架，但尚未验证最小策略问题
- 最近工作大多不能直接回答“这离初始目标更近了吗”
- 工作量很大，但无法明确说出当前阶段

## 5. 下一步建议原则

下一步建议必须满足：

- 能直接推进主链路
- 能减少目标不确定性
- 能作为阶段晋级前的明确里程碑
- 尽量是最小可交付闭环，而不是长期愿景

### 好的下一步

- 补齐 `project.yaml` 并明确研究入口
- 打通因子主表到标准化快照的闭环
- 为核心信号逻辑补一组失败测试和时间对齐校验
- 完成候选输出 schema 并做一次真实样本跑通

### 不好的下一步

- “继续完善系统”
- “进一步优化架构”
- “后续考虑统一平台”
- “可以把前端做得更完整”

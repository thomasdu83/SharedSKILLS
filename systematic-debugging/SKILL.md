# 系统化调试

调试时先找根因，再提修复方案。不要拿猜测代替证据。

## 何时使用

- 失败测试
- 生产 bug
- 集成异常
- 构建失败
- 性能问题
- 任何“看起来改一点就能好”的故障

## 铁律

```text
不做根因调查，不许提修复方案
```

## 四个阶段

| 阶段 | 关键动作 | 通过标准 |
|---|---|---|
| 根因调查 | 读错误、复现、检查近期变更、补诊断信息 | 知道问题发生在哪里、为什么发生 |
| 模式分析 | 找正常样例、对比差异、理解依赖 | 识别出真正不同点 |
| 假设验证 | 提出单一假设，最小化验证 | 假设被验证或被推翻 |
| 实施修复 | 先写失败测试，再做单一修复并验证 | 问题复现消失，测试通过 |

## 关键动作

### 根因调查

- 完整阅读错误、警告和堆栈
- 稳定复现问题；如果复现不了，继续收集数据，不要猜
- 检查近期代码、配置、依赖和环境变化
- 多组件系统先在边界加诊断信息，找出断裂层，而不是直接改最底层

### 模式分析

- 找一个能正常工作的同类实现
- 与参考实现逐项对比，不跳过看似很小的差异
- 搞清楚它依赖哪些配置、状态和前置条件

### 假设验证

- 一次只验证一个假设
- 一次只改一个变量
- 验证失败就回到调查阶段，不叠加更多补丁

### 实施修复

- 先写最小失败测试或复现脚本
- 只修已确认的根因，不顺手捆绑重构
- 修复后重新验证问题本身和受影响测试
- 如果 3 次以上修复都失败，暂停并质疑架构，而不是继续碰运气

## 不要这样做

- “先临时修一下，以后再排查”
- 一次改多个地方再跑测试
- 没有复现、没有证据就直接下结论
- 跳过测试只做手动验证
- 连续几次修复都没成功还继续套补丁

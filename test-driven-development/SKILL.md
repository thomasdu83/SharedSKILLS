# 测试驱动开发（TDD）

先写测试，看它失败，再写最少的代码让它通过。

## 何时使用

- 用户明确要求 TDD
- 高风险新行为、bug 修复、回归防护需要先有失败测试
- 计划长期维护的逻辑变更

## 何时先和用户确认

- 一次性原型
- 纯配置改动
- 明显不适合先写自动化测试的生成型产物

## 铁律

```text
没有失败的测试，就不写生产代码
```

如果已经先写了代码，再补测试不算 TDD。删除或回退到测试优先的顺序重新开始。

## 红-绿-重构

1. **红灯**：写一个最小失败测试，只测一个行为，名称清晰。
2. **验证红灯**：运行测试，确认它因预期原因失败，而不是拼写错误或环境错误。
3. **绿灯**：只写让该测试通过的最少代码，不顺手加功能。
4. **验证绿灯**：重新运行受影响测试，确认通过且没有额外异常。
5. **重构**：只在绿灯后清理命名、重复和结构，并保持测试继续为绿。

## 对测试的要求

- 一个测试只证明一个行为
- 优先测试真实行为，不优先测试 mock
- 测试应展示预期 API，而不是迎合当前实现
- 如果测试 setup 非常庞大，先怀疑设计复杂度

## 完成前检查

- [ ] 每个新行为或 bug 修复都有失败测试
- [ ] 在实现前亲眼看到测试失败
- [ ] 测试因预期原因失败
- [ ] 只写了让测试通过所需的最少代码
- [ ] 受影响测试全部通过
- [ ] 覆盖了边界情况和错误场景
